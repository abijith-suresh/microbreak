# AGENTS

This file defines rules for AI agents working on this repository.

## Document Ownership

| Document                | Owner       | Purpose                                                                                                                              |
| ----------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `README.md`             | Product     | User-facing only. Describes what the project is, what games are available, and the philosophy. No developer setup, no stack details. |
| `docs/CONTEXT.md`       | Product     | The single source of truth for what Microbreak is and isn't. Goals, non-goals, constraints, success criteria, game selection rules.  |
| `docs/ARCHITECTURE.md`  | Engineering | Technical truth. Stack, directory structure, game architecture pattern, persistence, styling, CI/CD.                                 |
| `docs/CONTRIBUTING.md`  | Engineering | Development workflow. Setup, scripts, quality gate, commit convention, how to add a new game.                                        |
| `AGENTS.md`             | Engineering | This file. Agent behavior rules and truth maintenance.                                                                               |
| `CHANGELOG.md`          | Automated   | Generated by release-please. Do not edit manually.                                                                                   |
| `src/pages/about.astro` | Product     | The public-facing philosophy page. Should stay aligned with CONTEXT.md.                                                              |

## Agent Behavior

### Before Making Changes

1. Read `docs/CONTEXT.md` to understand product boundaries
2. Read `docs/ARCHITECTURE.md` to understand patterns
3. Follow the existing game architecture pattern (engine → composable → session
   → app → page)
4. Never introduce server-side dependencies, external APIs, analytics, or
   tracking

### When Adding Features

- A feature must align with `docs/CONTEXT.md` goals and constraints
- If a feature violates a non-goal or hard constraint, flag it before
  implementing
- New games must follow the architecture pattern documented in
  `docs/ARCHITECTURE.md` and `docs/CONTRIBUTING.md`

### When Making Changes

- Run `bun run verify` before considering work complete
- Follow Conventional Commits format
- Do not edit `CHANGELOG.md` manually
- Use existing shared components (`GameScreen`, `ResultScreen`, `BackLink`,
  `Button`) rather than creating new ones

### Code Patterns

- Game logic stays in `src/lib/<game>.ts` (pure TypeScript, no framework)
- State management stays in `src/lib/<game>Game.ts` (SolidJS composable)
- Session validation uses type predicates in `src/lib/<game>Session.ts`
- UI components stay in `src/components/`
- New shared UI primitives go in `src/components/ui/`
- Home page preview components go in `src/components/previews/`
- Storage keys are centralized in `src/lib/storageKeys.ts`
- Always use the `storage.ts` abstraction for localStorage access

## Truth Maintenance

When product decisions change:

1. Update `docs/CONTEXT.md` first
2. Update `README.md` if the change is user-facing
3. Update `docs/ARCHITECTURE.md` if technical patterns change
4. Update `docs/CONTRIBUTING.md` if the workflow changes
5. Update `src/pages/about.astro` if the public philosophy changes

If you find a file that contradicts `docs/CONTEXT.md`, treat `CONTEXT.md` as
authoritative and flag the inconsistency for correction.
